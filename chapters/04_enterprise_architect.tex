\chapter{Enterprise Architect einrichten}

\begin{lernziele}
Nach diesem Kapitel können Sie die Rolle eines Modellierungswerkzeugs erklären, ein Enterprise-Architect-Projekt für das Roboterbeispiel strukturieren und grundlegende Modellierungsregeln anwenden. Sie wissen, welche Pakete für ein übersichtliches SysML-Modell sinnvoll sind und warum Namenskonventionen, Beschreibungen und Wiederverwendung von Modellelementen wichtig sind.
\end{lernziele}

\section{Einleitung}
SysML ist die Sprache, Enterprise Architect ist in diesem Buch das Werkzeug. Diese Unterscheidung ist wichtig. Ein Werkzeug macht aus einem schlechten Modell kein gutes Modell. Es kann aber helfen, Modellelemente zu verwalten, Diagramme zu erstellen, Beziehungen zu pflegen und Dokumentation zu erzeugen.

Enterprise Architect von Sparx Systems ist ein umfangreiches Modellierungswerkzeug. Es unterstützt SysML, UML und weitere Modellierungssprachen. Für Einsteiger wirkt die Oberfläche zunächst groß und etwas technisch. Das ist normal. Sie müssen am Anfang nicht jede Funktion verstehen. Für unser Lehrbuch reicht ein klarer, kleiner Arbeitsbereich.

\section{Rolle des Werkzeugs}
Enterprise Architect speichert nicht nur Zeichnungen. Es verwaltet Modellelemente. Ein Block, eine Anforderung oder ein Testfall kann in mehreren Diagrammen erscheinen und trotzdem dasselbe Element bleiben. Genau das macht das Werkzeug für MBSE nützlich.

Wichtige Aufgaben des Werkzeugs sind:

\begin{itemize}
    \item Modellelemente anlegen und verwalten
    \item Diagramme als Sichten auf das Modell erzeugen
    \item Beziehungen zwischen Elementen pflegen
    \item Anforderungen, Blöcke, Schnittstellen und Tests strukturieren
    \item Beschreibungen und Eigenschaften speichern
    \item später Dokumentation aus dem Modell erzeugen
\end{itemize}

\begin{hinweisbox}
Verwechseln Sie Werkzeugbedienung nicht mit Systems Engineering. Ein Klick erzeugt ein Element, aber die fachliche Entscheidung bleibt Ihre Aufgabe: Was bedeutet dieses Element? Warum existiert es? Womit hängt es zusammen?
\end{hinweisbox}

\section{Projekt anlegen}
Für das Lehrbuch verwenden wir ein Projekt mit dem Namen \texttt{AutonomousIndoorRobot}. Der Name ist bewusst englisch, weil spätere ROS2-Pakete und Nodes ebenfalls englische technische Bezeichnungen verwenden. Die erklärenden Texte im Modell können trotzdem deutsch sein.

Beim Anlegen des Projekts sollten Sie direkt eine einfache Startstruktur schaffen. Dadurch bleibt das Modell auch dann übersichtlich, wenn später viele Elemente hinzukommen.

\section{Empfohlene Paketstruktur}
Erstellen Sie folgende Pakete:

\begin{itemize}
    \item \texttt{00\_Overview}
    \item \texttt{01\_Stakeholders\_Context}
    \item \texttt{02\_Requirements}
    \item \texttt{03\_Structure}
    \item \texttt{04\_Behavior}
    \item \texttt{05\_Interfaces}
    \item \texttt{06\_Tests}
    \item \texttt{07\_ROS2\_Mapping}
    \item \texttt{08\_Decisions}
\end{itemize}

Diese Struktur ist bewusst schlicht. Sie trennt die wichtigsten Modellbereiche, ohne ein kompliziertes Metamodell einzuführen.

\subsection{00 Overview}
Dieses Paket enthält eine Startseite. Dort stehen Ziel des Modells, Systemgrenze, wichtige Annahmen und Links zu zentralen Diagrammen. Eine gute Startseite hilft besonders neuen Lesern.

\subsection{Stakeholders und Context}
Hier werden Stakeholder, externe Systeme und Kontextdiagramme abgelegt. Dazu gehören Nutzer, Wartung, Betreiber, Ladestation, Gebäude, Netzwerk und Personen im Umfeld.

\subsection{Requirements}
Dieses Paket enthält Anforderungen und Requirement Diagrams. Jede Anforderung erhält eine ID, einen verständlichen Text und bei Bedarf weitere Eigenschaften.

\subsection{Structure}
Hier liegen BDDs und strukturelle Blöcke. Dazu gehören \texttt{Robot}, \texttt{PerceptionSystem}, \texttt{NavigationSystem}, \texttt{BatterySystem} und weitere Systemelemente.

\subsection{Behavior}
Dieses Paket enthält Aktivitätsdiagramme und Zustandsdiagramme. Es beschreibt Abläufe und Betriebsmodi.

\subsection{Interfaces}
Hier werden Schnittstellen, Ports, Datenflüsse und später ROS2-nahe Nachrichtentypen dokumentiert.

\subsection{Tests}
Dieses Paket enthält Testfälle und Beziehungen zu Anforderungen. Es unterstützt die spätere Verifikation.

\subsection{ROS2 Mapping}
Hier wird dokumentiert, wie SysML-Blöcke, Funktionen und Schnittstellen auf ROS2-Nodes, Topics, Services und Actions abgebildet werden.

\section{Arbeitsweise im Modell}
Erstellen Sie Modellelemente möglichst nur einmal. Wenn Sie den Block \texttt{Camera} benötigen, legen Sie ihn einmal an und verwenden ihn in mehreren Diagrammen wieder. Ziehen Sie vorhandene Elemente aus dem Browser in neue Diagramme, statt sie immer neu zu erzeugen.

Diese Arbeitsweise ist am Anfang ungewohnt, aber sie ist zentral für MBSE. Nur so bleiben Beziehungen konsistent.

\section{Namenskonventionen}
Klare Namen machen Modelle lesbar. Für dieses Lehrbuch verwenden wir einfache Regeln:

\begin{itemize}
    \item Anforderungen: \texttt{REQ-001 Hindernisse erkennen}
    \item Testfälle: \texttt{TC-001 Navigation zu Zielpunkt}
    \item Blöcke: \texttt{PerceptionSystem}, \texttt{BatterySystem}
    \item ROS2-Nodes: \texttt{object\_detector\_node}
    \item Topics: \texttt{/camera/image\_raw}, \texttt{/battery\_state}
    \item Diagramme: \texttt{BDD Robot Overview}, \texttt{IBD Perception Data Flow}
\end{itemize}

Vermeiden Sie wechselnde Namen für dasselbe Element. Wenn ein Element einmal \texttt{BatterySystem} heißt, sollte es nicht an anderer Stelle \texttt{PowerModule} genannt werden, außer Sie modellieren bewusst unterschiedliche Dinge.

\section{Dokumentation im Modell}
Enterprise Architect erlaubt Beschreibungen direkt an Modellelementen. Nutzen Sie dieses Feld konsequent. Ein Diagramm ohne Beschreibung ist oft nur eine hübsche Zeichnung. Ein Modell mit klaren Beschreibungen bleibt auch Monate später verständlich.

Beschreiben Sie bei wichtigen Elementen:

\begin{itemize}
    \item Zweck
    \item Verantwortlichkeit
    \item wichtige Annahmen
    \item offene Fragen
    \item relevante Anforderungen
    \item Schnittstellen oder Abhängigkeiten
\end{itemize}

\section{Typische Anfängerfehler}
\subsection{Alles in ein Paket legen}
Wenn alle Elemente in einem Paket landen, wird das Modell schnell unübersichtlich. Nutzen Sie die Paketstruktur aktiv.

\subsection{Elemente mehrfach anlegen}
Mehrfach angelegte Elemente sehen ähnlich aus, sind aber nicht identisch. Dadurch brechen Traceability und Konsistenz.

\subsection{Diagramme ohne Zweck erstellen}
Jedes Diagramm sollte eine Frage beantworten. Wenn unklar ist, wozu ein Diagramm dient, ist es wahrscheinlich zu allgemein.

\subsection{Zu früh zu tief modellieren}
Modellieren Sie zuerst Systemkontext, Anforderungen, Hauptstruktur und zentrale Schnittstellen. Details können später folgen.

\section{Zusammenfassung}
Enterprise Architect ist das Werkzeug, mit dem wir das SysML-Modell des autonomen Indoor-Roboters aufbauen. Entscheidend ist eine einfache Projektstruktur, klare Namen, Wiederverwendung von Modellelementen und konsequente Dokumentation. Das Werkzeug unterstützt MBSE, ersetzt aber nicht das fachliche Denken.

\section{Übungen}
\begin{uebungbox}
\textbf{Übung 1: Paketstruktur anlegen}\\
Legen Sie die empfohlene Paketstruktur an und erstellen Sie eine Startseite im Modell. Beschreiben Sie dort Ziel, Umfang und Systemgrenze des Roboterprojekts.
\end{uebungbox}

\begin{uebungbox}
\textbf{Übung 2: Namensregeln anwenden}\\
Formulieren Sie Namen für fünf Anforderungen, drei Testfälle, fünf Blöcke und vier ROS2-Topics. Achten Sie auf einheitliche Schreibweise.
\end{uebungbox}

\begin{uebungbox}
\textbf{Übung 3: Diagrammzweck beschreiben}\\
Erstellen Sie eine Tabelle mit drei geplanten Diagrammen. Tragen Sie für jedes Diagramm ein, welche Frage es beantworten soll.
\end{uebungbox}
