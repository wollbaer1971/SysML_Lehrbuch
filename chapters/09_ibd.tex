\chapter{Verbindungsmodellierung mit \texttt{port} und \texttt{connection}}

\begin{lernziele}
Nach diesem Kapitel können Sie interne Verbindungen zwischen Systemteilen in SysML v2 beschreiben. Sie verstehen \texttt{port}, \texttt{connection} und Datenflüsse und können diese Konzepte auf ROS2-nahe Kommunikation im Roboterprojekt übertragen.
\end{lernziele}

\section{Einleitung}
Die Strukturmodellierung aus dem vorherigen Kapitel zeigt, welche Systemteile existieren. Die Verbindungsmodellierung zeigt, wie diese Teile zusammenarbeiten. Sie ist der nächste Schritt von der Struktur zur Zusammenarbeit.

In SysML v2 erfolgt die Verbindungsmodellierung mit \texttt{port} und \texttt{connection}. Ports beschreiben die Schnittstellen eines Systemteils nach außen. Verbindungen verknüpfen Ports verschiedener Teile miteinander.

\begin{hinweisbox}
SysML v1 verwendete das Internal Block Diagram (IBD) für diese Sicht. In SysML v2 sind Ports und Verbindungen direkte Modellelemente innerhalb von \texttt{part def}. Das Konzept bleibt ähnlich, die textuelle Notation ist präziser.
\end{hinweisbox}

\section{Zweck der Verbindungsmodellierung}
Eine Verbindungssicht beantwortet Fragen wie:

\begin{itemize}
    \item Welche Teile arbeiten innerhalb eines Systems zusammen?
    \item Welche Daten oder Signale fließen zwischen ihnen?
    \item Welche Ports besitzen die Teile?
    \item Welche Schnittstellen müssen beschrieben werden?
    \item Wo entstehen Integrationsrisiken?
\end{itemize}

\section{Ports}
Ein Port beschreibt einen Ein- oder Ausgang eines Systemteils. Ports machen Schnittstellen explizit und erzwingen, dass Daten- oder Signalflüsse benannt werden.

\begin{lstlisting}
port def ImagePort;
port def ScanPort;
port def ObjectListPort;

part def Camera {
  port imageOut : ImagePort;
}

part def Lidar {
  port scanOut : ScanPort;
}

part def ObjectDetector {
  port imageIn  : ~ImagePort;
  port objectsOut : ObjectListPort;
}

part def ObstacleDetector {
  port scanIn : ~ScanPort;
}
\end{lstlisting}

Das Tilde-Zeichen \texttt{\textasciitilde} markiert einen \glsdisp{conjport}{konjugierten Port}, der empfängt statt sendet.

\section{Verbindungen}
\texttt{connection} verknüpft kompatible Ports miteinander. Eine Verbindung sollte immer dokumentieren, was über sie fließt. Die Grundform lautet: \texttt{connection <Name> connect <Teil1>.<Port1> to <Teil2>.<Port2>;}

\begin{lstlisting}
part def PerceptionSystem {
  part camera          : Camera;
  part lidar           : Lidar;
  part objectDetector  : ObjectDetector;
  part obstacleDetector : ObstacleDetector;

  connection c1 connect camera.imageOut
                      to objectDetector.imageIn;
  connection c2 connect lidar.scanOut
                      to obstacleDetector.scanIn;
}
\end{lstlisting}

\section{Datenflüsse und ROS2}
Die Verbindungsmodellierung passt gut zur ROS2-Kommunikation. ROS2-Topics können als Datenflüsse modelliert werden. Ein Topic \texttt{/camera/image\_raw} verbindet \texttt{camera\_node} und \texttt{object\_detector\_node}. Ein Topic \texttt{/cmd\_vel} verbindet Navigation und Bewegungssteuerung.

\begin{praxisbox}
Eine Verbindungssicht für das \texttt{PerceptionSystem} zeigt: Die Kamera liefert Bilddaten an den \texttt{ObjectDetector}, während der Lidar Laserscandaten an den \texttt{ObstacleDetector} liefert. Beide Ergebnisse gehen an die Navigation. Daraus lassen sich später ROS2-Topics direkt ableiten.
\end{praxisbox}

\section{Verbindungsmodell für die Navigation}
Eine Verbindungssicht für die Navigation macht sichtbar, dass Lokalisierung, Karte, Pfadplanung und Bewegungssteuerung zusammenarbeiten:

\begin{lstlisting}
part def NavigationSystem {
  part localization : Localization;
  part mapProvider  : MapProvider;
  part pathPlanner  : PathPlanner;
  part pathFollower : PathFollower;

  connection c1 connect localization.poseOut
                      to pathPlanner.poseIn;
  connection c2 connect mapProvider.mapOut
                      to pathPlanner.mapIn;
  connection c3 connect pathPlanner.pathOut
                      to pathFollower.pathIn;
}
\end{lstlisting}

Dabei liefert \texttt{c1} die aktuelle Pose der Lokalisierung an den Pfadplaner, \texttt{c2} liefert die Karte an denselben Pfadplaner, und \texttt{c3} gibt den geplanten Pfad an die Bewegungssteuerung weiter.

\section{Annahmen dokumentieren}
Ein gutes Verbindungsmodell sollte auch Annahmen dokumentieren. Wird die Karte vorher geladen oder während der Fahrt aktualisiert? Arbeitet die Lokalisierung relativ zu \texttt{map} oder \texttt{odom} (zwei gebräuchliche ROS2-Koordinatensysteme, die in Kapitel 10 näher erklärt werden)? Solche Fragen gehören als Dokumentation (\texttt{doc /* ... */}) zum Modellelement.

\begin{lstlisting}
part def NavigationSystem {
  doc /* Voraussetzung: Eine initiale Karte ist vorhanden.
         Lokalisierung arbeitet relativ zum map-Frame. */
  ...
}
\end{lstlisting}

\section{Typische Fehler}
\subsection{Nur Teile verbinden ohne Ports}
Verbindungen sollten immer über benannte Ports laufen. Direkte Verbindungen ohne Ports lassen offen, was genau übertragen wird.

\subsection{Ports ohne Datentypen}
Ein Port wird erst nützlich, wenn bekannt ist, welche Daten darüber fließen. Definieren Sie \texttt{port def} mit sprechenden Namen.

\subsection{Zu viele Details in einem Modellabschnitt}
Eine Verbindungssicht sollte eine klare Frage beantworten. Für große Systeme sind mehrere fokussierte Abschnitte besser als ein überladenes Modell.

\section{Zusammenfassung}
Die Verbindungsmodellierung mit \texttt{port} und \texttt{connection} zeigt die interne Zusammenarbeit von Systemteilen. Sie ist besonders nützlich für Schnittstellen und Datenflüsse. Im Roboterprojekt bildet sie eine Brücke von der SysML-v2-Struktur zur ROS2-Kommunikation.

\section{Übungen}
\begin{uebungbox}
\textbf{Übung 1: Verbindungsmodell PerceptionSystem}\\
Erstellen Sie eine \texttt{part def PerceptionSystem} mit Kamera, Lidar, ObjectDetector und ObstacleDetector. Definieren Sie Ports und verbinden Sie sie mit \texttt{connection}.
\end{uebungbox}

\begin{uebungbox}
\textbf{Übung 2: Ports benennen}\\
Definieren Sie für \texttt{BatterySystem}, \texttt{NavigationSystem} und \texttt{MotionControl} jeweils mindestens zwei Ports mit sprechenden Namen.
\end{uebungbox}

\begin{uebungbox}
\textbf{Übung 3: ROS2-Mapping vorbereiten}\\
Wählen Sie drei Verbindungen aus Ihrem Modell und ordnen Sie ihnen jeweils ein mögliches ROS2-Topic zu.
\end{uebungbox}
