\chapter{Werkzeuge für SysML v2 einrichten}

\begin{lernziele}
Nach diesem Kapitel können Sie erklären, warum dieses Lehrbuch eine textuelle SysML-v2-Arbeitsweise verwendet. Sie können ein einfaches Projekt in VS Code mit dem Syside Editor anlegen und SysML-v2-Dateien strukturieren. Außerdem können Sie Ziel, Umfang und Systemgrenze, Anforderungen, Part Definitions, Testfälle und ROS2-Topic-Zuordnungen in der SysML-v2-Textnotation formulieren.
\end{lernziele}

\section{Einleitung}
SysML v2 kann grafisch und textuell verwendet werden. Für dieses Lehrbuch steht die textuelle Notation im Vordergrund. Das passt gut zu modernen Entwicklungsabläufen: Modelle liegen in Dateien, können versioniert, verglichen, überprüft und gemeinsam mit Code gepflegt werden.

Als kostenloser Einstieg eignet sich der Syside Editor für VS Code. Er unterstützt SysML-v2-Dateien mit Syntaxhervorhebung, Autovervollständigung, Formatierung, Gliederung und Validierung. Für grafische Sichten kann zusätzlich Eclipse SysON genutzt werden. SysON ist ein webbasiertes Open-Source-Werkzeug für SysML-v2-Modelle.

\begin{hinweisbox}
Ein Werkzeug löst keine Modellierungsaufgabe automatisch. Es hilft beim Schreiben, Prüfen und Anzeigen des Modells. Die fachliche Entscheidung bleibt Ihre Aufgabe: Welche Elemente gibt es? Welche Beziehungen sind wichtig? Welche Anforderungen müssen nachverfolgbar sein?
\end{hinweisbox}

\section{Warum textuelle Modellierung?}
Viele Einsteiger erwarten bei SysML zuerst Diagramme. Diagramme bleiben wichtig, aber SysML v2 macht den Modellinhalt stärker als präzisen Text zugänglich. Ein Modell kann dadurch ähnlich behandelt werden wie Quellcode.

Textuelle Modellierung hat mehrere Vorteile:

\begin{itemize}
    \item Modelle lassen sich in Git versionieren.
    \item Änderungen sind als Textvergleich sichtbar.
    \item Beispiele können direkt im Lehrbuch gezeigt werden.
    \item Modellteile können leichter wiederverwendet werden.
    \item Validierung und Automatisierung werden einfacher.
    \item ROS2-Code und SysML-v2-Modell können im selben Projekt liegen.
\end{itemize}

Das bedeutet nicht, dass Diagramme unwichtig werden. Diagramme sind weiterhin wertvolle Sichten. Der Unterschied ist: Das Modell entsteht nicht nur durch Zeichnen, sondern durch präzise definierte Modellelemente.

\section{Ziel, Umfang und Systemgrenze beschreiben}

Die erste Datei \texttt{00\_overview.sysml} dient dazu, das Projekt zu rahmen. Sie beantwortet drei grundlegende Fragen:

\begin{itemize}
    \item \textbf{Ziel:} Wofür ist das System da?
    \item \textbf{Umfang:} Was gehört dazu, was nicht?
    \item \textbf{Systemgrenze:} Wo endet das System und wo beginnt die Umgebung?
\end{itemize}

In SysML v2 beginnt jede Datei mit einem \texttt{package}. Das Package ist der benannte Modellraum. Innerhalb davon beschreiben wir Ziel und Umfang als Dokumentation mit \texttt{doc~/*~...~*/}. Die Systemgrenze zeigt sich darin, was wir als \texttt{part def} definieren und was wir als externen Akteur behandeln.

Beachten Sie: In \texttt{doc}-Kommentaren schreiben wir Umlaute bewusst aus (\texttt{ae}, \texttt{oe}, \texttt{ue}), um Encoding-Probleme zwischen unterschiedlichen Werkzeugen und Betriebssystemen zu vermeiden.

\begin{lstlisting}
package AutonomousIndoorRobot {
  doc
  /* Ziel: Ein autonomer Indoor-Roboter transportiert kleine
     Gegenstaende innerhalb eines Gebaeudes zu ausgewaehlten Zielen.
     Er faehrt selbststaendig, erkennt Hindernisse und kehrt bei
     niedrigem Akkustand zur Ladestation zurueck.

     Umfang: Dieses Modell beschreibt Navigation, Wahrnehmung,
     Energieversorgung, Bedienung und Sicherheit des Roboters.

     Systemgrenze: Zur Ladestation und zum Gebaeude gehoert die
     physische Infrastruktur, die der Roboter nutzt, aber nicht
     selbst steuert. Personen sind externe Akteure. */

  part def Robot;
  part def ChargingStation;  // extern, aber modellrelevant
}
\end{lstlisting}

Die \texttt{part def Robot} zeigt: Das System ist der Roboter. Die \texttt{ChargingStation} taucht als externe Komponente auf, weil der Roboter sie anfahren muss, sie aber nicht Teil des Roboters ist. Diese Entscheidung muss bewusst getroffen und dokumentiert werden.

\begin{praxisbox}
Häufiger Anfängerfehler: Systemgrenze unklar lassen. Wenn Sie nicht entscheiden, ob die Ladestation \enquote{im System} oder \enquote{außen} ist, entstehen später Widersprüche in Anforderungen und Tests. Eine kurze \texttt{doc}-Notiz am Anfang verhindert das.
\end{praxisbox}

Diese Darstellung ist für den Einstieg vereinfacht: \texttt{ChargingStation} steht hier als \texttt{part def} auf derselben Ebene wie \texttt{Robot}. Sauberer wäre es, externe Systeme in einem eigenen Kontext-Package zu modellieren, um die Systemgrenze auch strukturell sichtbar zu machen.

\section{Anforderungen formulieren}

Anforderungen beschreiben, was das System leisten muss, nicht wie. In SysML v2 werden sie in der Datei \texttt{02\_requirements.sysml} mit dem Schlüsselwort \texttt{requirement} definiert.

Jede Anforderung bekommt einen eindeutigen Namen und eine Beschreibung als \texttt{doc}-Block. Anforderungen können geschachtelt werden, um Hierarchien auszudrücken.

\begin{lstlisting}
package Requirements {

  requirement REQ_Navigate {
    doc /* Der Roboter muss in einem bekannten Innenraumbereich
           autonom ein ausgewaehltes Ziel anfahren koennen. */
  }

  requirement REQ_Safety {
    doc /* Sicherheitsanforderungen fuer den Betrieb
           in der Naehe von Personen. */

    requirement REQ_DetectObstacles {
      doc /* Der Roboter muss Hindernisse im Fahrweg erkennen. */
    }

    requirement REQ_StopSafely {
      doc /* Der Roboter muss bei einem Hindernis in 0,5 m
             Abstand bremsen und bei 0,2 m stoppen. */
    }
  }

  requirement REQ_Battery {
    doc /* Akkumanagement und Energieversorgung. */

    requirement REQ_MonitorBattery {
      doc /* Der Roboter muss seinen Akkustand kontinuierlich
             ueberwachen. */
    }

    requirement REQ_ReturnToDock {
      doc /* Der Roboter muss bei einem Akkustand unter 20 Prozent
             selbststaendig zur Ladestation fahren. */
    }
  }
}
\end{lstlisting}

Diese Schachtelung ist zunächst eine reine Namensraum-Gruppierung: Sie ordnet verwandte Anforderungen thematisch unter einer übergeordneten Anforderung ein, erzeugt aber keine automatische \texttt{derive}-Beziehung zwischen übergeordneter und untergeordneter Anforderung; eine solche Ableitungsbeziehung müssten Sie bei Bedarf zusätzlich explizit mit \texttt{derive} modellieren.

Lesen Sie jede Anforderung laut vor und fragen Sie: Kann ich das testen? Wenn nicht, ist sie zu vage. \enquote{Der Roboter muss navigieren} ist nicht testbar. \enquote{Der Roboter muss bei 20 Prozent Akkustand die Ladestation anfahren} ist testbar.

\section{Part Definitions formulieren}

Part Definitions beschreiben die Bausteine des Systems. In SysML v2 definiert \texttt{part def} einen Typ. \texttt{part} erzeugt eine Instanz davon. Die Datei \texttt{03\_structure.sysml} enthält die Strukturmodellierung.

Beginnen Sie mit dem Hauptelement \texttt{Robot} und listen Sie seine Hauptteile auf. Jeder Teil wird zuerst grob benannt. Die Details kommen schrittweise.

\begin{lstlisting}
package Structure {

  part def Robot {
    doc /* Das Gesamtsystem. Enthaelt alle funktionalen Subsysteme. */

    part perception  : PerceptionSystem;
    part navigation  : NavigationSystem;
    part motion      : MotionControl;
    part battery     : BatterySystem;
    part ui          : UserInterface;
    part safety      : SafetyLogic;
  }

  part def PerceptionSystem {
    doc /* Nimmt Sensordaten auf und verarbeitet sie zu
           Objektlisten und Hindernissignalen. */

    part frontCamera : Camera;
    part lidar       : Lidar;
    part detector    : ObjectDetector;
  }

  part def BatterySystem {
    doc /* Ueberwacht den Akkustand und meldet kritische Werte. */

    part cell    : Battery;
    part monitor : BatteryMonitor;
    part charger : ChargingInterface;
  }

  part def NavigationSystem;
  part def MotionControl;
  part def UserInterface;
  part def SafetyLogic;
  part def Camera;
  part def Lidar;
  part def ObjectDetector;
  part def Battery;
  part def BatteryMonitor;
  part def ChargingInterface;
}
\end{lstlisting}

\section{Testfälle anlegen}

Testfälle überprüfen, ob das System eine Anforderung erfüllt. Mit \texttt{verification def} wird der Testfall selbst definiert, mit dem Schlüsselwort \texttt{verify} wird anschließend die Verbindung zwischen diesem Testfall und einer Anforderung ausgedrückt. Die Datei \texttt{06\_tests.sysml} enthält die Testmodellierung.

Ein Testfall beschreibt: Was wird getestet? Welche Vorbedingungen gelten? Was ist die erwartete Reaktion? Welche Anforderung wird damit geprüft?

\begin{lstlisting}
package Tests {

  verification def TC_BatteryLowReturn {
    doc /* Test: Roboter kehrt bei niedrigem Akkustand zurueck.
         Vorbedingung: Roboter navigiert, Akkustand betraegt 25 Prozent.
         Aktion: Akkustand sinkt auf 18 Prozent.
         Erwartung: Roboter bricht Navigation ab und faehrt Ladestation an.
         Akzeptanzkriterium: Ladestation erreicht innerhalb von 120 Sekunden. */
  }

  verification def TC_ObstacleStop {
    doc /* Test: Roboter haelt vor Hindernis an.
         Vorbedingung: Roboter faehrt mit 0,3 m/s.
         Aktion: Hindernis erscheint in 0,4 m Abstand.
         Erwartung: Roboter bremst und stoppt innerhalb von 0,2 m Sicherheitsabstand.
         Akzeptanzkriterium: Keine Kollision, Stopp in weniger als 2 Sekunden. */
  }

  part robot : Robot {
    verify REQ_ReturnToDock    by tc1 : TC_BatteryLowReturn;
    verify REQ_StopSafely      by tc2 : TC_ObstacleStop;
  }
}
\end{lstlisting}

Die \texttt{verify}-Beziehung verbindet den Testfall mit der Anforderung. Das ist der Kern der Nachvollziehbarkeit: Jede Anforderung sollte mindestens einen Testfall haben, der sie prüft. Anforderungen ohne Test bleiben im Projekt unkontrolliert.

Die genaue Syntax von \texttt{verify ... by} kann sich zwischen SysML-v2-Werkzeugversionen leicht unterscheiden, da die Sprache noch in Entwicklung ist. Prüfen Sie im Zweifel gegen Ihre installierte Werkzeugversion.

\section{ROS2-Topics zuordnen}

Die Datei \texttt{07\_ros2\_mapping.sysml} verbindet das SysML-Modell mit der späteren ROS2-Implementierung. Für jeden Systemteil wird dokumentiert, welcher ROS2-Node ihn umsetzt und welche Topics er verwendet.

In SysML v2 können solche Zuordnungen als Attribute oder als Dokumentation modelliert werden. Das Schlüsselwort \texttt{attribute} definiert dabei eine benannte, typisierte Eigenschaft eines Modellelements, hier jeweils vom Typ \texttt{String}. Für den Einstieg reicht eine klare Textnotation:

\begin{lstlisting}
package ROS2Mapping {

  part def BatterySystem_ROS2 {
    doc /* ROS2-Umsetzung des BatterySystem.

         Node: battery_monitor_node
         Paket: autonomous_indoor_robot/battery_monitor

         Publizierte Topics:
           /battery_state   (sensor_msgs/BatteryState)  -- Akkustand, 1 Hz

         Abonnierte Topics:
           keine

         Verbindung zum Modell:
           BatterySystem.monitor -> battery_monitor_node
           REQ_MonitorBattery wird durch diesen Node erfuellt. */

    attribute rosNode  : String = "battery_monitor_node";
    attribute topicOut : String = "/battery_state";
  }

  part def NavigationSystem_ROS2 {
    doc /* ROS2-Umsetzung des NavigationSystem.

         Nodes: nav2_stack (extern), path_planner_node
         Abonnierte Topics:
           /battery_state   -- triggert Rueckkehr bei niedrigem Akku
           /scan            (sensor_msgs/LaserScan)  -- Lidardaten
           /map             (nav_msgs/OccupancyGrid) -- Karte

         Publizierte Topics:
           /cmd_vel         (geometry_msgs/Twist) -- Fahrbefehle
           /robot_status    (std_msgs/String)     -- Betriebszustand

         Verbindung zum Modell:
           REQ_Navigate und REQ_ReturnToDock werden durch diese Nodes erfuellt. */

    attribute rosNode  : String = "nav2_stack";
    attribute topicIn  : String = "/battery_state, /scan, /map";
    attribute topicOut : String = "/cmd_vel, /robot_status";
  }
}
\end{lstlisting}

Für den Einstieg vereinfacht: Mehrere Topics in einem String zusammenzufassen ist praktisch, aber unsauber. Sauberer wäre ein eigenes Attribut je Topic (z.\,B. \texttt{topicScan}, \texttt{topicMap}), damit jedes Topic einzeln typisiert und nachverfolgt werden kann.

Diese Datei wird im Laufe des Projekts wachsen. Beginnen Sie mit den Nodes, die Sie als Erstes implementieren. Die Verbindung zwischen SysML-Systemelement und ROS2-Node sollte immer explizit dokumentiert sein.

Diese Struktur ist bewusst schlicht. Sie trennt die wichtigsten Modellbereiche, ohne schon zu Beginn ein kompliziertes Metamodell einzuführen.

\section{Namenskonventionen}
Klare Namen machen Modelle lesbar. Für dieses Lehrbuch verwenden wir einfache Regeln:

\begin{itemize}
    \item Packages: \texttt{AutonomousIndoorRobot}
    \item Part Definitions: \texttt{Robot}, \texttt{BatterySystem}, \texttt{NavigationSystem}
    \item Parts (Instanzen): \texttt{battery}, \texttt{navigation}, \texttt{motion}
    \item Anforderungen: \texttt{REQ\_DetectObstacles}, \texttt{REQ\_ReturnToDock}
    \item Testfälle: \texttt{TC\_BatteryLowReturn}, \texttt{TC\_ObstacleStop}
    \item ROS2-Nodes: \texttt{battery\_monitor\_node}, \texttt{path\_planner\_node}
    \item ROS2-Topics: \texttt{/battery\_state}, \texttt{/cmd\_vel}, \texttt{/robot\_status}
\end{itemize}

Vermeiden Sie wechselnde Namen für dasselbe Element. Wenn ein Element einmal \texttt{BatterySystem} heißt, sollte es nicht an anderer Stelle \texttt{PowerModule} genannt werden, außer Sie modellieren bewusst unterschiedliche Dinge.

\section{Grafische Sichten}
Grafische Sichten sind besonders hilfreich, wenn ein Team schnell über Struktur oder Schnittstellen sprechen möchte. In SysML v2 sollte man sie als Darstellung des Modells verstehen, nicht als Ersatz für das Modell.

Eclipse SysON kann für solche Sichten interessant sein. Es ist besonders dann nützlich, wenn Sie aus dem textuellen Modell heraus Struktur-, Verbindungs- oder Anforderungssichten visualisieren wollen. Für den Einstieg in diesem Buch reicht es aber, zunächst die textuelle Notation sauber zu verstehen.

Wichtig ist dabei die Blickrichtung: Der Text bleibt die Quelle der Wahrheit, die Grafik ist eine automatisch erzeugte Sicht darauf. Wenn Sie im Syside Editor eine neue \texttt{part} in \texttt{Robot} ergänzen und speichern, kann SysON daraus die passende Struktursicht neu erzeugen, ohne dass Sie selbst einen Kasten zeichnen müssen. \section{Typische Anfängerfehler}
\subsection{Systemgrenze nicht klären}
Viele Modelle beginnen ohne eine klare Antwort auf die Frage: Was gehört zum System? Diese Entscheidung prägt alle späteren Anforderungen und Tests. Dokumentieren Sie die Systemgrenze explizit in der Übersichtsdatei.

\subsection{Anforderungen nicht testbar formulieren}
Anforderungen wie \enquote{Der Roboter soll gut navigieren} sind nicht prüfbar. Formulieren Sie immer mit konkreten Bedingungen und messbaren Erwartungen.

\subsection{Part Definitions ohne Verantwortung}
Eine \texttt{part def} ohne \texttt{doc}-Block lässt offen, wozu dieser Baustein da ist. Schreiben Sie für jeden Baustein mindestens einen Satz über seine Aufgabe.

\subsection{Testfälle ohne Verbindung zu Anforderungen}
Ein Testfall, der keine Anforderung prüft, ist nutzlos für die Nachvollziehbarkeit. Jede \texttt{verification def} sollte mindestens eine \texttt{verify}-Beziehung haben.

\subsection{ROS2-Topics nicht dokumentieren}
Der Name eines Topics allein reicht nicht. Welcher Node sendet? Welcher empfängt? Welches Nachrichtenformat? Diese Angaben gehören in die Mapping-Datei.

\subsection{Text wie Programmcode ohne Bedeutung schreiben}
SysML-v2-Text sieht teilweise wie Code aus. Trotzdem modellieren Sie kein Programm, sondern ein System. Fragen Sie bei jedem Element: Welche fachliche Aussage steckt dahinter?

\section{Zusammenfassung}
Dieses Kapitel hat gezeigt, wie ein SysML-v2-Projekt von Anfang an strukturiert wird. Ziel, Umfang und Systemgrenze werden im Übersichtspaket als \texttt{doc}-Blöcke festgehalten. Anforderungen werden als \texttt{requirement} mit prüfbarem Text formuliert. Part Definitions beschreiben die Systembausteine mit \texttt{part def}. Testfälle verbinden sich über \texttt{verify} mit Anforderungen. ROS2-Topics werden in einer Mapping-Datei als \texttt{attribute} und Dokumentation zugeordnet. VS Code mit dem Syside Editor ist die empfohlene kostenlose Arbeitsumgebung. Entscheidend bleibt nicht das Werkzeug, sondern ein verständliches, konsistentes und nachverfolgbares Modell.

\section{Übungen}
