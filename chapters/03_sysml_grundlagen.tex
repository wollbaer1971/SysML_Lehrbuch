\chapter{SysML-Grundlagen}

\begin{lernziele}
Nach diesem Kapitel kennen Sie die Grundidee von \gls{sysml} v2, die wichtigsten Modellierungskonzepte und deren Rolle in einem MBSE-Projekt. Sie können erklären, warum ein SysML-Diagramm nur eine Sicht auf ein Modell ist, welche Konzepte für den Einstieg besonders wichtig sind und wie diese Konzepte beim autonomen Indoor-Roboter zusammenspielen.
\end{lernziele}

\section{Einleitung}
In den ersten beiden Kapiteln haben wir geklärt, warum Systems Engineering und MBSE nützlich sind. Jetzt benötigen wir eine Sprache, mit der wir Anforderungen, Struktur, Verhalten, Schnittstellen und Tests präzise beschreiben können. Diese Sprache ist in diesem Buch \gls{sysml} v2.

\gls{sysml} steht für Systems Modeling Language. Die Sprache wurde entwickelt, um technische Systeme zu modellieren. SysML v2 ist eine vollständige Neuentwicklung gegenüber SysML v1 mit einer präzisen textuellen Notation und einer klaren, modernen Semantik. Sie ist stärker auf Systeme aus Hardware, Software, Menschen, Prozessen, Daten und physikalischen Zusammenhängen ausgerichtet als UML. Das passt gut zu unserem autonomen Indoor-Roboter: Er besteht nicht nur aus Software, sondern auch aus Sensorik, Antrieb, Batterie, Gehäuse, Ladestation, Bedienung und realer Umgebung.

Für Einsteiger ist wichtig: SysML ist kein Zeichenprogramm. SysML ist eine Modellierungssprache. Ein Diagramm ist nur eine sichtbare Darstellung von Modellelementen. Der eigentliche Wert entsteht im Modell: Anforderungen sind mit Systemteilen verbunden, Systemteile besitzen Schnittstellen, Verhalten beschreibt Abläufe, Tests verifizieren Anforderungen.

\section{Warum eine Modellierungssprache?}
Ohne gemeinsame Sprache entstehen in Projekten schnell Missverständnisse. Ein Teammitglied zeichnet Kästen für Funktionen, ein anderes für Softwaremodule, ein drittes für Hardwarekomponenten. Alle Diagramme sehen ähnlich aus, meinen aber unterschiedliche Dinge. SysML hilft, solche Bedeutungen zu klären.

SysML v2 bietet vordefinierte Konzepte:

\begin{itemize}
    \item \texttt{requirement} für Anforderungen
    \item \texttt{part def} und \texttt{part} für Systemelemente und deren Instanzen
    \item \texttt{port} und \texttt{connection} für Schnittstellen
    \item \texttt{action def} und \texttt{action} für Abläufe
    \item \texttt{state def} und \texttt{state} für Zustände
    \item \texttt{calc def} und \texttt{constraint} für mathematische Zusammenhänge
    \item \texttt{satisfy}, \texttt{verify}, \texttt{derive} für Nachvollziehbarkeit
\end{itemize}

Damit kann ein Team über dasselbe System sprechen, ohne jedes Symbol neu erklären zu müssen.

\begin{praxisbox}
Wenn im Modell eine \texttt{part def PerceptionSystem} existiert, kann sie in einer Struktursicht als Teil des Roboters erscheinen, in einer Verbindungssicht mit Kamera und Lidar verknüpft werden und über eine \texttt{satisfy}-Beziehung Anforderungen zur Hinderniserkennung erfüllen. Es ist derselbe Modellinhalt, nur aus verschiedenen Blickwinkeln.
\end{praxisbox}

\section{Wichtige Konzepte in SysML v2}
SysML v2 kennt keine festen Diagrammtypen mehr wie in v1. Stattdessen gibt es Sichten (\emph{views}) und Sichtpunkte (\emph{viewpoints}) auf ein gemeinsames Modell. Ein Sichtpunkt legt fest, welche Art von Frage eine Sicht beantworten soll, zum Beispiel Struktur oder Verhalten; die Sicht selbst ist die daraus erzeugte konkrete Darstellung. Für den Einstieg konzentrieren wir uns auf diejenigen Konzepte, die für das Roboterprojekt den größten Nutzen bringen. \subsection{Anforderungen (\texttt{requirement})}
Anforderungen beschreiben, was das System leisten muss. In SysML v2 werden sie textuell mit \texttt{requirement} definiert. Beziehungen wie \texttt{satisfy}, \texttt{verify} und \texttt{derive} verbinden Anforderungen mit Systemteilen und Testfällen.

Typische Fragen:

\begin{itemize}
    \item Was muss das System leisten?
    \item Welche Anforderungen hängen zusammen?
    \item Welche Anforderung wurde aus welcher höheren abgeleitet?
    \item Welche Teile erfüllen eine Anforderung?
    \item Welche Tests überprüfen eine Anforderung?
\end{itemize}

\subsection{Strukturmodellierung (\texttt{part def}, \texttt{part})}
In SysML v2 beschreibt \texttt{part def} den Typ eines Systemelements. \texttt{part} erzeugt eine Instanz davon. Zusammen ersetzen sie den Block-Begriff aus SysML v1.

\begin{lstlisting}
package RobotStructure {
  part def Robot {
    part perceptionSystem : PerceptionSystem;
    part navigationSystem : NavigationSystem;
    part batterySystem    : BatterySystem;
    part motionControl    : MotionControl;
  }
  part def PerceptionSystem;
  part def NavigationSystem;
  part def BatterySystem;
  part def MotionControl;
}
\end{lstlisting}

Diese Darstellung beantwortet die Frage: Aus welchen Hauptbausteinen besteht das System?

\subsection{Verbindungsmodellierung (\texttt{port}, \texttt{connection})}
\texttt{port} beschreibt die Schnittstelle eines Systemteils nach außen. \texttt{connection} verbindet Ports verschiedener Teile miteinander. Damit wird sichtbar, wie Teile eines Systems intern zusammenarbeiten.

\begin{lstlisting}
part def PerceptionSystem {
  port sensorData : ~SensorDataPort;
  part camera  : Camera;
  part lidar   : Lidar;
  part detector : ObjectDetector;
}
\end{lstlisting}

\begin{hinweisbox}
Das Tilde-Zeichen (\texttt{\textasciitilde}) kennzeichnet einen sogenannten konjugierten Port; was das genau bedeutet, erklären wir ausführlich in Kapitel 9.
\end{hinweisbox}

\subsection{Verhaltensmodellierung (\texttt{action def}, \texttt{action})}
Aktivitäten beschreiben Abläufe. Sie eignen sich für Prozesse wie \enquote{Transportauftrag ausführen}, \enquote{Zur Ladestation zurückkehren} oder \enquote{Hindernis behandeln}. In SysML v2 werden Abläufe mit \texttt{action def} und \texttt{action} modelliert.

\subsection{Zustandsmodellierung (\texttt{state def}, \texttt{state})}
Zustandsmodelle beschreiben Zustände und Übergänge. Ein Roboter besitzt Betriebszustände wie \texttt{Idle}, \texttt{Navigating}, \texttt{AvoidingObstacle} und \texttt{Charging}. In SysML v2 werden diese mit \texttt{state def} definiert.

\subsection{Parametrische Modellierung (\texttt{calc def}, \texttt{constraint})}
Parametrische Modelle beschreiben mathematische Beziehungen. Sie helfen, technische Entscheidungen früh zu prüfen. In SysML v2 werden Berechnungen mit \texttt{calc def} und Bedingungen mit \texttt{constraint} ausgedrückt.

Verhalten, Zustände und Parametrik lernen Sie ab Kapitel 11 im Detail kennen. Für den ersten Einstieg in Kapitel 4 genügen zunächst Anforderungen und Struktur.

\section{Modell statt Bild}
Ein SysML-Diagramm ist nur eine Sicht auf ein Modell. Diese Unterscheidung ist zentral. Wenn Sie in einem Zeichenprogramm drei Kästen mit dem Namen \texttt{NavigationSystem} zeichnen, sind das drei unabhängige grafische Objekte. In einem Modellierungswerkzeug wie dem Syside Editor ist \texttt{NavigationSystem} ein einziges Modellelement, das in mehreren Sichten dargestellt werden kann.

Das hat praktische Vorteile:

\begin{itemize}
    \item Namen bleiben konsistent.
    \item Beziehungen können ausgewertet werden.
    \item Änderungen sind leichter nachvollziehbar.
    \item Verschiedene Sichten zeigen denselben Inhalt.
    \item Traceability wird möglich.
\end{itemize}

\section{Wie die Konzepte zusammenarbeiten}
SysML-v2-Modellelemente sollten nicht isoliert entstehen. Sie ergänzen sich.

Beim Indoor-Roboter kann eine Modellkette so aussehen:

\begin{enumerate}
    \item Im Anforderungsmodell steht: \enquote{Der Roboter muss Hindernisse erkennen.}
    \item \texttt{part def PerceptionSystem} wird als Systemteil definiert.
    \item Ports und Verbindungen zeigen, wie Kamera und Lidar Daten liefern.
    \item Eine \texttt{action def} beschreibt, wie ein Hindernis während der Fahrt behandelt wird.
    \item Ein \texttt{state def} modelliert, dass der Roboter bei Gefahr in einen Stopp- oder Ausweichzustand wechselt.
    \item Ein Testfall wird über \texttt{verify} mit der Anforderung verbunden.
\end{enumerate}

So entsteht ein zusammenhängendes Modell statt einer losen Sammlung schöner Bilder. Abbildung~\ref{fig:kap03-struktur-verbindungssicht} zeigt beispielhaft, wie Struktur- und Verbindungssicht des \texttt{PerceptionSystem} zusammenspielen: Die Übersicht zeigt die Subsysteme von \texttt{Robot}, die Detailsicht zeigt \texttt{PerceptionSystem} mit seinen Teilen und dem Port \texttt{sensorData}.

\section{SysML v1 und SysML v2 im Vergleich}
Einsteiger stoßen in der Literatur häufig noch auf SysML v1. Die wichtigsten Begriffsänderungen sind:

\begin{center}
\adjustbox{max width=\linewidth}{%
\begin{tabular}{ll}
\toprule
\textbf{SysML v1} & \textbf{SysML v2} \\
\midrule
Block (block) & \texttt{part def} \\
Block Definition Diagram (BDD) & Struktursicht (\texttt{part def}, \texttt{part}) \\
Internal Block Diagram (IBD) & Verbindungssicht (\texttt{port}, \texttt{connection}) \\
Activity Diagram & \texttt{action def}, \texttt{action} \\
State Machine Diagram & \texttt{state def}, \texttt{state} \\
Parametric Diagram & \texttt{calc def}, \texttt{constraint} \\
\texttt{deriveRqt} & \texttt{derive} \\
\bottomrule
\end{tabular}%
}
\end{center}

Dieses Buch verwendet durchgehend SysML v2. Wo alte Begriffe aus Kompatibilitätsgründen erwähnt werden, ist das explizit gekennzeichnet.

\section{Typische Anfängerfehler}
\subsection{Zu viele Elemente zu früh}
Einsteiger möchten oft sofort alle Konzepte verwenden. Das führt schnell zu Überforderung. Beginnen Sie mit Anforderungen, \texttt{part def} und einfachen Verbindungen. Verhalten und Parametrik können danach folgen.

\subsection{Unklare Bedeutung von Systemteilen}
Manchmal werden Funktionen, Softwaremodule, Hardwareteile und Rollen wild gemischt. Das ist nicht grundsätzlich verboten, muss aber bewusst geschehen. Eine \texttt{part def Camera} ist ein physisches Element. Eine \texttt{part def ObjectDetector} ist eher eine Software- oder Funktionskomponente.

\subsection{Diagramme ohne Text}
Ein Diagramm ohne Beschreibung ist oft schwer verständlich. Nutzen Sie Dokumentationsfelder (\texttt{doc /* ... */}). Schreiben Sie kurz, welchen Zweck ein Modellelement hat.

\subsection{Keine Wiederverwendung}
Wenn dasselbe Element mehrfach neu angelegt wird, verliert das Modell seinen Wert. Definieren Sie Typen mit \texttt{part def} einmal und instanziieren Sie sie mit \texttt{part}.

\section{Zusammenfassung}
SysML v2 ist eine Modellierungssprache für technische Systeme. Sie hilft, Anforderungen, Struktur, Verhalten, Schnittstellen und Nachweise einheitlich zu beschreiben. Für dieses Buch sind besonders \texttt{requirement}, \texttt{part def}, \texttt{port}, \texttt{connection}, \texttt{action def}, \texttt{state def} und \texttt{calc def} wichtig. Entscheidend ist nicht das Zeichnen einzelner Diagramme, sondern der Aufbau eines zusammenhängenden Modells.

\section{Übungen}
\begin{uebungbox}
\textbf{Übung 1: Konzept zuordnen}\\
Ordnen Sie folgende Fragen einem SysML-v2-Konzept zu: Was muss das System leisten? Aus welchen Teilen besteht es? Welche Daten fließen zwischen Komponenten? Welche Zustände kann der Roboter haben? Wie läuft ein Transportauftrag ab?
\end{uebungbox}

\begin{uebungbox}
\textbf{Übung 2: Erste Modellkette}\\
Wählen Sie die Funktion \enquote{Akkustand überwachen}. Beschreiben Sie dazu eine Anforderung, eine \texttt{part def}, einen Port, einen möglichen Zustand und einen Testfall.
\end{uebungbox}

\begin{uebungbox}
\textbf{Übung 3: v1 vs. v2}\\
Nennen Sie für BDD, IBD und Constraint Block jeweils das entsprechende SysML-v2-Konzept und erklären Sie in einem Satz den Unterschied.
\end{uebungbox}
